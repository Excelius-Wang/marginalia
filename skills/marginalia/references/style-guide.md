# 写作风格与信息来源纪律（Style Guide）

## 输出语言

默认中文。专业术语保留英文原词，例如 attention、fine-tuning、ablation、retrieval、policy gradient、representation、benchmark、threat model、MEV，不要机械翻译。

## 信息来源纪律（四类）

整个分析与写作过程中，严格区分四类信息，并在表述里让读者能分辨：

- **论文原文明确声称**：作者显式 claim 或展示了这一点。
- **相关文献中的已有结论**：现有文献里的记录结果，尽可能引用来源。
- **基于证据的合理推断**：从论文展示的内容逻辑推出，但未被显式声明——要标成推断。
- **仍然不确定的猜测**：你的假设，没有直接证据支撑——要标成猜测。

不要把推断写成事实，不要把猜测写成推断。不确定某个 claim 属于哪一类，直接说出来。评估发表后影响、采用、后续工作时，用当前这一轮检索到的来源，不要凭记忆。

## 风格锚点

技术写作的自然度参考 Andrej Karpathy，技术表达的清晰度参考 Kaiming He。只学可迁移的技术写作特征：

- 从具体的技术情境出发，不要悬空总结。
- 直接点名问题，先展示失败模式，再解释方法。
- 用具体名词、动作和数字，少用抽象形容词。
- 先写 claim，再写 evidence。
- 保留不确定性，不要装作什么都确定。

## 具体要求

- 每句话都要有信息量，避免大空话和泛泛而谈。
- 以流畅段落为主，只有在结构确实有帮助时才用 bullet（如实验设计、pipeline 步骤、最小复现）。
- 避免滥用破折号、引号、括号、分号。
- 避免低信息量句式，例如"不是 A 而是 B""重要的是""值得注意的是""总而言之"。
- 不要用 groundbreaking、革命性 这类夸张词；novelty 按论文自己的 framing 描述。
- 把"论文声称"和"我的判断"分开表述。

## 语义加粗与彩色标记（发布到飞书会渲染成颜色）

写笔记时，像一个编辑一样**主动判断**哪里该加粗、哪个词该上什么颜色——这是写作的一部分，不要留到发布时再补，也不要机械地见词就标。本地 md 仍可读；发布脚本把这些标记渲染成飞书里的彩色文本（见 `feishu-publish.md`）。

**加粗 `**…**`**：只加在一句话的**核心论断主干**上（每条 bullet 的判断句、关键取舍、结论），不是整句、不是修饰语。

**彩色标记 `{{c:词}}`**：不同颜色承担不同含义，全篇保持一致。`c` 取值与作用：

| 标记 | 颜色 | 作用 | 例 |
| --- | --- | --- | --- |
| `{{b:…}}` | 蓝 | **专名实体**：模型 / 方法 / 系统 / 数据集 / benchmark | `{{b:Qwen2.5-VL}}`、`{{b:GenEval}}` |
| `{{p:…}}` | 紫 | **机制与技术概念** | `{{p:flow matching}}`、`{{p:MaPE}}`、`{{p:双专家通道}}` |
| `{{o:…}}` | 橙 | **关键指标 / 数字结论** | `{{o:0.90}}`、`{{o:VBench 85.11}}` |
| `{{g:…}}` | 绿 | **正面判断 / 优势 / 有效推进** | `{{g:同类最佳}}` |
| `{{r:…}}` | 红 | **风险 / 短板 / 批判 / 最脆弱假设** | `{{r:证据偏薄}}`、`{{r:营销话术}}` |

纪律（避免变花）：

- **克制**：只标真正承载信息的词——专名首次出现、关键指标、批判落点；同一个词不必每次都标。一段里彩色词控制在几个以内。
- **颜色不串味**：蓝只给专名、红只给批判，别拿红色当普通强调。颜色和加粗可叠加（`{{r:**…**}}`）表示"既是批判落点又是论断主干"。
- 颜色是**语义**不是装饰：读者应能靠颜色快速扫出"这篇用了哪些模型(蓝)、哪些机制(紫)、考了多少分(橙)、强在哪(绿)、弱在哪(红)"。
